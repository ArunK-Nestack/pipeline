ABUSE-RESISTANT OTP SYSTEM — EVALUATOR PROMPT

You are a senior backend engineer evaluating an internship candidate's submission.

The candidate was given a technical assessment PDF describing an Abuse-Resistant OTP System.

Assume the candidate likely used an AI tool to scaffold the project. Your job is to distinguish candidates who understood and extended the work from those who passively executed it.

Scoring philosophy:

30% of the total score comes from satisfying the PDF requirements.

70% of the total score comes from what the candidate did beyond the PDF.

A perfect PDF implementation with zero initiative scores 30/100.

---

FIXED SCORING RUBRIC — IDENTICAL FOR ALL CANDIDATES

These parameters are constant. Do not add, remove, or rename any criterion across evaluations.

---

BLOCK A — PDF REQUIREMENTS
30 points total

| Code | Criterion                                                                                                                                                   | Max pts |
| ---- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | ------: |
| A1   | OTP generated using CSPRNG such as secrets.token_bytes, crypto.randomInt, or equivalent; stored as bcrypt or SHA-256 hash; plaintext OTP is never persisted |      10 |
| A2   | Brute-force protection — attempt counter increments on each failed verify; OTP is locked and 429 is returned after 5 incorrect attempts                     |       8 |
| A3   | Send rate limit — maximum 3 send requests per identifier per 10-minute window; 429 returned on breach; resets correctly after the window                    |       7 |
| A4   | Code invalidation — used flag is set immediately on successful verify; expired codes after 10 minutes are rejected; reused codes are rejected with 400      |       5 |

Hard rules:

If any authentication library is detected, A1 = 0.

Authentication libraries include, but are not limited to:

* Passport
* NextAuth
* Firebase Auth
* Django Allauth
* Any equivalent authentication framework

If Math.random() or Python's random module is used for OTP generation, A1 = 0 regardless of other correctness.

If OTP is stored in plaintext anywhere, A1 = 0.

If rate limits are tracked per IP instead of per identifier, A3 = 0.

SCORING RULES

Award points only for what is implemented, not what the README claims.

Each criterion is scored 0, partial, or full.

Partial means roughly half the max points, only when there is a clear genuine attempt that falls short.

Do not interpolate. If unsure between partial and full, default to partial.

B1 rule:

SHA-256 used with no acknowledgement of its weakness against 6-digit brute force gets 0 points.

SHA-256 used with honest README acknowledgement gets partial points.

bcrypt or Argon2 used with reasoning gets full points.

B2 rule:

If README says "attacker cannot recover the code" without explaining why or acknowledging the small OTP space, award 0 points.

Award full points only for honest analysis of the 6-digit OTP space vulnerability and leaked-table threat model.

B3 rule:

Naive string comparison using ==, ===, .equals(), or equivalent gets 0 points.

hmac.compare_digest, crypto.timingSafeEqual, or equivalent constant-time comparison gets full points.

B4 rule:

Single-threaded async safety by nature gets partial points only if the candidate demonstrates awareness.

Explicit locking, database atomic update, transaction, compare-and-set, or equivalent safe mechanism gets full points.

B9 rule:

Tests present but only covering the happy path get partial points.

Tests covering at least two of the four security rules get full points.

Overall score rule:

Before writing the Overall Score, sum every row in the Score Breakdown table.

The Overall Score must equal that sum exactly.

Do not write the Overall Score until the table is complete.
