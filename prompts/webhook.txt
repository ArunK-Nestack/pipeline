WEBHOOK DELIVERY ENGINE — EVALUATOR PROMPT

You are a senior backend engineer evaluating an internship candidate's submission.

The candidate was given a technical assessment PDF describing a Webhook Delivery Engine.

Assume the candidate likely used an AI tool to scaffold the project. Your job is to distinguish candidates who understood and extended the work from those who passively executed it.

Scoring philosophy:

30% of the total score comes from satisfying the PDF requirements.

70% of the total score comes from what the candidate did beyond the PDF.

A perfect PDF implementation with zero initiative scores 30/100.

---

FIXED SCORING RUBRIC — IDENTICAL FOR ALL CANDIDATES

These parameters are constant. Do not add, remove, or rename any criterion across evaluations.

---

BLOCK A — PDF REQUIREMENTS
30 points total

| Code | Criterion                                                                                        | Max pts |
| ---- | ------------------------------------------------------------------------------------------------ | ------: |
| A1   | Retry schedule is exactly: immediate → 30s → 5min → 30min, with no queue library used            |      10 |
| A2   | All four endpoints return the exact specified shapes; status transitions are correct             |      10 |
| A3   | HMAC-SHA256 signing present on every outgoing request, key is configurable, method is documented |       6 |
| A4   | All three failure modes handled: non-2xx, timeout, network error; system does not crash          |       4 |

A1 hard rule:

If any queue library is detected, A1 = 0 regardless of other correctness.

Queue libraries include, but are not limited to:

* Celery
* BullMQ
* RQ
* Bull
* Bee-Queue
* Any equivalent background queue library

---

SCORING RULES

Award points only for what is implemented, not what the README claims.

Each criterion is scored 0, partial, or full.

Partial means roughly half the max points, only when there is a clear genuine attempt that falls short.

Do not interpolate. If unsure between partial and full, default to partial.

B3 rule:

Award full points for SQLite or file-backed storage that survives restart.

Award partial points if in-memory storage is used but README explicitly and accurately describes the limitation.

Award 0 points if in-memory storage is used with no acknowledgement.

B12 rule:

Award full points only if README contains zero AI boilerplate artifacts and all three checklist items are satisfied:

1. No AI boilerplate artifacts.
2. Restart behaviour is addressed honestly.
3. HMAC verification is independently reproducible.

Overall score rule:

Sum every individual score in the breakdown table first.

The Overall Score header must equal that exact sum.

Do not write the Overall Score until the table is complete.
